# Library Management API

Backend-проект для учебной библиотеки, разработанный студентами на Java + Spring Boot. Цель проекта — отработать командную разработку, API-first подход, работу с БД и деплой на тестовый стенд.

## Цели проекта

- Научить студентов проектировать REST API и бизнес-логику.
- Разделить роли: аналитики, разработчики, тестировщики, DevOps.
- Собрать рабочий **MVP**: каталог книг, читатели, выдачи, бронирования.
- Научиться выкатывать приложение в Docker на тестовый стенд.

## Функциональность MVP

### Каталог книг

- Создание, просмотр, редактирование, удаление книг.
- Поиск по названию, автору, жанру.
- Фильтрация по доступности (есть ли свободные экземпляры).

### Читатели

- Регистрация читателя.
- Просмотр информации о читателе.
- Ограничение на максимальное количество активных выдач.

### Выдачи (Loans)

- Выдача книги читателю.
- Возврат книги.
- Запрет выдачи, если нет доступных экземпляров или превышен лимит.

### Бронирования (Reservations)

- Бронирование книги при отсутствии доступных экземпляров.
- Отмена брони.

## Технологический стек

- **Язык:** Java 17+
- **Фреймворк:** Spring Boot
- **Модули:** Spring Web, Spring Data JPA, Validation
- **База данных:** PostgreSQL
- **Документация API:** Springdoc OpenAPI / Swagger UI
- **Контейнеризация:** Docker, Docker Compose
- **Тестирование:** JUnit, Spring Test, Postman (ручные тесты)

## Архитектура проекта

Типичная слоистая архитектура:

- `controller` — REST-контроллеры, принимают HTTP-запросы и возвращают DTO.
- `service` — бизнес-логика (правила выдачи, бронирования и т.п.).
- `repository` — доступ к БД через Spring Data JPA.
- `domain/model` — сущности JPA.
- `dto` — объекты для запросов/ответов API.
- `config` — конфигурация приложения (profiles, Swagger и т.д.).

## Основные сущности

- `Book` — запись о книге (название, авторы, ISBN, жанр, всего экземпляров, доступных экземпляров).
- `Author` — автор книги.
- `Reader` — читатель библиотеки.
- `Loan` — факт выдачи книги читателю.
- `Reservation` — бронь книги.

## Основные endpoint’ы

Примеры (URI могут отличаться в финальной реализации):

### Каталог

- `GET /api/books` — список книг с фильтрами и пагинацией.
- `GET /api/books/{id}` — подробная информация о книге.
- `POST /api/books` — создать книгу.
- `PUT /api/books/{id}` — обновить книгу.
- `DELETE /api/books/{id}` — удалить книгу.

### Читатели

- `GET /api/readers` — список читателей.
- `GET /api/readers/{id}` — информация о читателе.
- `POST /api/readers` — регистрация читателя.

### Выдачи

- `POST /api/loans` — оформить выдачу книги.
- `POST /api/loans/{id}/return` — вернуть книгу.
- `GET /api/loans` — список выдач (для библиотекаря).

### Бронирования

- `POST /api/reservations` — создать бронь.
- `DELETE /api/reservations/{id}` — отменить бронь.

Полный контракт описан в OpenAPI-спецификации (`openapi.yaml`) или доступен в Swagger UI.

## Бизнес-правила

Примеры ключевых правил:

- Нельзя выдать книгу, если нет доступных экземпляров.
- Читатель не может иметь больше N активных выдач.
- Нельзя удалить книгу, если по ней есть активные выдачи.
- Бронь возможна только если нет доступных экземпляров книги.
- При возврате книги количество доступных экземпляров увеличивается.

## Запуск проекта локально

1. Установить:
   - JDK 17+
   - Docker и Docker Compose
   - PostgreSQL (если запуск без Docker)

2. Склонировать репозиторий:

```bash
git clone <URL-репозитория>
cd library-management-api
```

3. Запуск через Docker Compose (рекомендуется):

```bash
docker compose up --build
```

4. Запуск без Docker:

- Настроить БД в `application-dev.yml`.
- Собрать и запустить:

```bash
./mvnw spring-boot:run
```

5. Проверка API:

- Swagger UI: `http://localhost:<port>/swagger-ui.html` или `/swagger-ui/index.html`.
- Либо использовать Postman-коллекцию (если она есть в репозитории).

## Профили окружений

- `dev` — локальная разработка.
- `test` / `docker` — тестовый стенд с Docker Compose.

Профиль выбирается через `SPRING_PROFILES_ACTIVE`.

## Роли в команде

- **Аналитики** — описывают требования, user story, бизнес-правила, API-контракт.
- **Backend-разработчики** — реализуют контроллеры, сервисы, сущности, тесты.
- **QA** — пишут тестовые сценарии, проверяют API.
- **DevOps-lite** — Docker, профили и деплой на тестовый стенд.
- **Тимлид** — план, ревью кода, интеграция и релизы.

## Как контрибьютить

- Каждая задача — отдельная ветка: `feature/<task-name>`.
- Перед началом работы синхронизировать ветку с `develop`/`main`.
- Создать Pull Request с описанием задачи.
- Пройти code review и исправить замечания.
